# Pillog — 식약처 공공데이터 API 조사 보고서

## 개요

Pillog 서비스의 핵심 데이터 소스가 될 식약처(식품의약품안전처) 공공 데이터 API를 조사했습니다. 건강기능식품의 제품 정보, 성분/원재료, 영양 정보를 제공하는 API들을 정리하고, Pillog에서의 활용 방안을 함께 제시합니다.

---

## 1. 핵심 API 목록

### 1-1. 건강기능식품 품목제조신고(원재료) — `C003`

| 항목 | 내용 |
|------|------|
| **제공 기관** | 식품의약품안전처 (식품안전나라) |
| **서비스 코드** | `C003` |
| **Base URL** | `http://openapi.foodsafetykorea.go.kr/api/{인증키}/C003/{json\|xml}/{시작}/{끝}` |
| **호출 제한** | 1일 1,000건 |
| **업데이트 주기** | 상시 |
| **Pillog 중요도** | ⭐⭐⭐⭐⭐ (핵심) |

**제공 필드:**

| 필드명 | 변수명 | Pillog 활용 |
|--------|--------|-------------|
| 인허가번호 | `LCNS_NO` | 업체 고유 식별 |
| 업소명 | `BSSH_NM` | 제조사 정보 표시 |
| 품목제조번호 | `PRDLST_REPORT_NO` | 제품 고유 식별자 (PK) |
| 품목명 | `PRDLST_NM` | 제품 검색/표시 |
| 보고일자 | `PRMS_DT` | 신규/변경 추적 |
| 소비기한 | `POG_DAYCNT` | 유통기한 안내 |
| 성상 | `DISPOS` | 제품 형태 정보 |
| 섭취방법 | `NTK_MTHD` | 복용 가이드 |
| **주된 기능성** | `PRIMARY_FNCLTY` | **맞춤 추천의 핵심 데이터** |
| **섭취 시 주의사항** | `IFTKN_ATNT_MATR_CN` | **상호작용/안전성 체크** |
| 보관방법 | `CSTDY_MTHD` | 보관 안내 |
| 기준규격 | `STDR_STND` | 성분 함량 기준 |
| **원재료** | `RAWMTRL_NM` | **성분 분석의 핵심 데이터** |
| 제품형태 | `PRDT_SHAP_CD_NM` | 필터링용 |

**요청 파라미터 (선택):**

| 변수명 | 설명 | 활용 |
|--------|------|------|
| `PRDLST_NM` | 품목명으로 검색 | 사용자 제품 검색 |
| `BSSH_NM` | 업소명으로 검색 | 브랜드별 조회 |
| `PRDLST_REPORT_NO` | 품목제조번호로 검색 | 상세 정보 조회 |
| `CHNG_DT` | 변경일자 이후 데이터 | 데이터 동기화 (증분 업데이트) |
| `LCNS_NO` | 인허가번호로 검색 | 업체별 제품 조회 |

**샘플 요청:**
```
http://openapi.foodsafetykorea.go.kr/api/{KEY}/C003/json/1/5/PRDLST_NM=홍삼
```

## 2. 인증키 발급 방법

### 식품안전나라 (C003 등)
1. `https://www.foodsafetykorea.go.kr` 회원가입
2. 데이터활용서비스 → 인증키 관리 → 인증키 신청
3. 원하는 API 서비스 선택 후 이용 목적 입력
4. 승인 후 즉시 사용 가능

### 공공데이터포털 (data.go.kr)
1. `https://www.data.go.kr` 회원가입
2. 원하는 데이터셋 페이지에서 "활용신청"
3. 활용 목적 입력 후 신청
4. 자동 승인 또는 1~2일 내 승인

> 💡 **Tip:** 두 포털 모두 무료이며, 동일 데이터가 양쪽에 있을 수 있으므로 호출 한도 분산 가능

## 3. Pillog 활용 전략

### 3-1. 데이터 수집 파이프라인

```
[GitHub Actions (크론)]
        │
        ▼
[Python 스크립트]
   ├─ 식품안전나라 C003 API 호출 (전체 제품 + 원재료)
   ├─ 건강기능식품 영양DB API 호출 (원료 분류)
   └─ 변경일자(CHNG_DT) 기반 증분 업데이트
        │
        ▼
[데이터 정제/파싱]
   ├─ 원재료(RAWMTRL_NM) 파싱 → 개별 성분 추출
   ├─ 주된기능성(PRIMARY_FNCLTY) 파싱 → 기능성 태그 분류
   └─ 주의사항(IFTKN_ATNT_MATR_CN) 파싱 → 상호작용 경고 추출
        │
        ▼
[Supabase PostgreSQL 저장]
```

### 3-2. 원재료 데이터 파싱 예시

C003 API의 `RAWMTRL_NM` 필드는 쉼표로 구분된 원재료 목록입니다:

```
홍삼농축액(농축물),시클로덱스트린시럽,액상프락토올리고당(고형분기준 55%),정제수,...
```

**파싱 전략:**
- 쉼표 기준 분리
- 괄호 내 부가 정보 별도 저장 (함량, 비율 등)
- 기능성 원료 vs 부형제/첨가물 분류
- 원료명 정규화 (동일 원료의 다른 표기 통합)

### 3-3. 기능성 데이터 파싱 예시

`PRIMARY_FNCLTY` 필드 구조:

```
[홍삼제품]①면역력 증진②피로회복③혈소판 응집 억제를 통한 혈액흐름에 도움④기억력 개선
```

**파싱 전략:**
- `[카테고리명]` 추출 → 기능성 원료 분류
- 원문자 번호(①②③) 기준 분리 → 개별 기능성 효과 추출
- 기능성 효과 → 사용자 건강 고민 매핑 테이블 구축

### 3-4. 주요 기능별 API 매핑

| Pillog 기능 | 주요 API | 활용 필드 |
|-------------|----------|-----------|
| 제품 검색 | C003 | PRDLST_NM, BSSH_NM |
| 성분 분석 | C003 + 영양DB | RAWMTRL_NM, STDR_STND |
| 기능성 분류 | C003 | PRIMARY_FNCLTY |
| 복용 가이드 | C003 | NTK_MTHD, IFTKN_ATNT_MATR_CN |
| 안전성 경고 | C003 | IFTKN_ATNT_MATR_CN |
| 맞춤 추천 | C003 + 영양DB | PRIMARY_FNCLTY + 원료 분류 |

## 4. 기술적 고려사항

### 4-1. API 호출 제한 대응

- **C003 기준 1일 1,000건 제한** → 전체 데이터 초기 수집 시 여러 날에 걸쳐 분산 호출
- 한 번 요청에 최대 1,000건까지 가능 (`startIdx=1&endIdx=1000`)
- `CHNG_DT` 파라미터로 변경분만 증분 동기화 → 일일 호출량 최소화

### 4-2. 데이터 품질 이슈

- 원재료명 표기가 업체마다 상이 (예: "비타민C" vs "L-아스코르빈산" vs "비타민 C")
- 기능성 텍스트 포맷이 불규칙 → 정규표현식 + 예외 처리 필요
- 일부 레거시 데이터는 필드가 비어있을 수 있음

### 4-3. 보완 데이터 소스

| 소스 | 용도 | URL |
|------|------|-----|
| NIH Dietary Supplement Label DB | 해외 제품/성분 참조 | dsld.od.nih.gov |
| 식품안전나라 식품영양성분DB | 일반 식품 영양정보 | various.foodsafetykorea.go.kr |
| 의약품 낱알식별 정보 | 의약품-영양제 상호작용 참조 | data.go.kr/data/15057639 |
